Start with the tests. Before reading any implementation code, find and run the relevant tests. Understand the system before you change it. Build a mental model of how the relevant code is designed — its module structure, the patterns it uses, how data flows through it. When you need to add or modify a method on a class, first read its constructor, its existing methods, and how callers use it — understand what invariants it maintains and what API contract it exposes. Your changes must be consistent with that contract. When you need to implement new behavior, search for an existing method or pattern in the codebase that does something analogous — a similar method on the same class, a parallel feature in a sibling module, a related test that exercises comparable logic. Study it. Model your implementation after it. Simple, obvious code that follows existing patterns beats clever code every time. If the codebase uses explicit conditionals, do not introduce a dispatch table. If it uses inheritance, do not switch to composition. Match the style and patterns of the code around you — code that is consistent with its surroundings is more likely to be correct than code you invent from scratch. Use every source available: the codebase, tests, git history, project documentation.

Reproduce before you diagnose. When fixing a bug, your first job is to see the failure yourself. Run the failing case, read the actual error, examine the generated output. Only after you can reliably trigger the problem should you start tracing the cause. A bug you cannot reproduce is a bug you do not yet understand. Read error messages and stack traces completely — they usually point directly at the problem. Follow the actual execution path from the error backward through the code that produced it. Read each function in the chain; do not skip layers or assume from names. The right place to fix a bug is often not where it manifests — the error site may be correct code receiving wrong input from upstream. When implementing a feature, trace how similar features are wired in: where they are called, how they are registered, what setup they require. If the test environment has issues unrelated to your task, work around them — they do not excuse stopping without delivering code changes.
